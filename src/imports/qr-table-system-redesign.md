현재 내가 만든 QR테이블 시스템 프론트엔드 시안을 기준으로, 단순히 보기 좋게 리디자인하지 말고 실제 상용 SaaS 수준의 구조로 재정리해줘.

프로젝트 전제:
- QR테이블 주문 시스템
- 현재는 프론트엔드 프로토타입 단계
- 이후 서버 연결 예정
- 결제 기능은 초기 버전에서는 제외
- 핵심 범위는 고객 주문 / 관리자 운영 / 웨이팅 / QR 진입 구조
- 목표는 식당 대표와 실사용 운영자가 봤을 때 실제 서비스처럼 신뢰감 있게 보이도록 만드는 것

현재 포함된 화면/기능:
- 랜딩 페이지
- 고객용 모바일 QR 주문 화면
- 관리자 대시보드/운영 화면
- 웨이팅 등록 화면
- 메뉴 탐색
- 옵션 선택
- 장바구니
- 주문하기
- 직원 호출
- 주문 상태 확인
- 관리자 주문 현황
- 테이블 현황
- 웨이팅 관리
- 메뉴 관리
- QR 관리 구조 일부

중요 전제:
- 지금 단계에서는 백엔드 완성 여부가 아니라 프론트 구조 완성도를 높이는 것이 목적
- 결제/정산/PG 연동은 지금 단계에서 제외
- 디자인보다 운영 UX와 정보 구조가 더 중요함
- 고객 화면은 B2C스럽게 부드럽고 친숙해도 되지만
- 관리자 화면은 B2B SaaS/POS처럼 정보 밀도와 가독성이 우선이어야 함

반드시 아래 기준으로 분석하고 재구성해줘:

1. 고객 주문 화면
- 메뉴 탐색이 직관적인지
- 스크롤 중에도 카테고리 이동이 쉬운지
- 메뉴 → 옵션 → 장바구니 → 주문 흐름이 가장 선명하게 보이는지
- 이벤트, 주문내역, 호출 등 보조 기능이 메인 주문 흐름을 방해하지 않는지
- 하단 장바구니 CTA가 수량/총액/주문 버튼까지 포함한 플로팅 바 또는 바텀시트 구조로 더 명확해질 수 있는지

2. 관리자 운영 화면
- 예쁜 화면이 아니라 실제 매장 피크타임 운영에 적합한지
- 한 화면에 더 많은 정보를 실수 없이 빠르게 읽을 수 있는지
- 주문 상태 / 테이블 상태 / 웨이팅 상태가 색상과 뱃지 규칙으로 통일되어 있는지
- 지나치게 둥글거나 여백이 커서 정보 밀도가 떨어지지 않는지
- 핵심 KPI가 화면 최상단에서 즉시 보이는지
- 현재 관리자 화면을 역할별로 분리해야 하는지

3. 화면 역할 분리
현재 관리자 영역을 다음 구조로 나누는 방향으로 제안해줘:
- /admin/pos : 카운터/홀 운영용 테이블 현황 및 주문 처리
- /admin/kds : 주방용 주문 티켓 및 상태 변경
- /admin/stats : 매출/운영 통계
- /admin/waiting : 웨이팅 운영 관리
이렇게 분리했을 때 가장 좋은 레이아웃과 UI 구조를 제안해줘

4. 웨이팅 화면
- 모바일 웨이팅과 키오스크 웨이팅이 분리되어야 하는지
- /waiting 과 /waiting/kiosk 를 어떻게 다르게 설계하면 좋은지
- 현장 등록용 태블릿/키오스크에 맞는 큰 버튼, 큰 입력 폼, 높은 가독성 구조를 제안해줘

5. 라우팅 구조
아래 구조를 기준으로 더 나은 구조가 있다면 개선해줘:
- /
- /qr/:tableId 또는 /qr/:token
- /waiting
- /waiting/kiosk
- /admin/pos
- /admin/kds
- /admin/stats
- /admin/waiting

상태와 운영 시각언어가 통일되도록 아래 항목을 포함해 제안해줘:

8. 실무적 목표
- 프론트만 봐도 서비스가 허술해 보이지 않게
- 투자자/대표/점주가 봤을 때 실제 운영을 이해하고 만든 느낌이 나게
- 나중에 서버/API를 붙여도 갈아엎지 않도록 구조적으로 설계
- 너무 화려한 UI보다 실제 운영성과 신뢰감이 느껴지게
- 고객용은 친절하고 쉬워야 하고, 관리자용은 빠르고 명확해야 함

출력 형식은 아래 순서로 정리해줘:

1. 현재 구조에서 잘된 점
2. 현재 구조에서 문제점
3. 지금 가장 먼저 수정해야 할 10가지
4. 고객 주문 화면 개선안
5. 관리자 POS 화면 개선안
6. KDS 화면 신설안
7. 웨이팅 모바일 / 키오스크 분리안
8. 라우팅 구조 최종안
9. 피그마 컴포넌트 구조안
10. 디자인 토큰 최종안
11. 지금 단계에서 바로 적용할 것 / 나중에 적용할 것으로 구분한 우선순위표

중요:
- 단순히 “더 예쁘게”가 아니라 “더 운영 친화적이고 제품 구조적으로 완성도 있게” 제안해줘
- 관리자 화면은 B2C 감성이 아니라 실무형 POS/KDS 관점으로 분석해줘
- 테이블 맵(Floor plan)은 멋진 제안일 수 있지만, 초기 단계에서 과한 경우라면 보류해도 된다고 판단해줘
- 지금 당장 중요한 것은 라우팅 분리, 상태 규칙 통일, 컴포넌트 분리, 고객 주문 흐름 단순화라는 점을 반영해줘
