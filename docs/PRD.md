# PRD: 테이블QR (TableFlow) — SaaS 전환

**문서 버전:** 1.0
**작성일:** 2026-03-14
**상태:** 초안

---

## 1. 개요 (Overview)

### 배경

현재 테이블QR은 단일 레스토랑용 React SPA 프로토타입입니다. 모든 데이터가 하드코딩된 Mock 데이터이며, 백엔드·인증·멀티테넌시가 없는 상태입니다.

### 목표

외식업 자영업자·소상공인을 대상으로 **QR 주문 + POS + 대기 관리**를 제공하는 B2B SaaS 플랫폼으로 전환합니다.

### 핵심 가치 제안

- 별도 앱 설치 없이 QR 스캔 → 웹 기반 주문
- 점주는 어디서나 실시간 주문·테이블·매출 확인
- 월 구독 요금만으로 포스기 없이 운영 가능

| **프랜차이즈 관리자** | 다점포 브랜드 오너 | 점포별 통합 현황, 브랜드 메뉴 일괄 관리 |

## 3. 사용자 여정 (User Journey)

### 3-1. 점주 온보딩

```
회원가입 → 매장 정보 입력 → 플랜 선택 →
메뉴 등록 → 테이블 설정 → QR 코드 생성/출력 → 운영 시작
```

### 3-2. 고객 주문 흐름

```
QR 스캔 → 테이블 자동 인식 → 메뉴 탐색/검색 →
옵션 선택 → 장바구니 → 주문 제출 → 실시간 상태 추적
```

### 3-3. 점주 일상 운영

```
대시보드 접속 → 실시간 주문 알림 → 상태 업데이트 →
테이블 현황 확인 → 당일 매출 요약
```

## 5. 기술 아키텍처 (Technical Architecture)

### 5-1. 시스템 구성

```
[고객 브라우저]          [점주 브라우저]
      │                       │
      └──────────┬────────────┘
                 │ HTTPS
        [Frontend — React SPA]
        현재 코드베이스 유지·확장
                 │
         [API Gateway]
                 │
    ┌────────────┼────────────┐
    │            │            │
[REST API]  [WebSocket]  [Auth Service]
    │            │
[PostgreSQL] [Redis Pub/Sub]
    │
[File Storage — 메뉴 이미지]
(S3 / Cloudflare R2)
```

### 5-2. 기술 스택 선택

| 레이어 | 현재 | SaaS 전환 후 |
|-------|------|------------|
| Frontend | React + Vite (유지) | + React Query, Zustand, Socket.IO client |
| Backend | 없음 | **Hono.js** (Bun/Node) 또는 **Next.js API Routes** |
| Database | 없음 | **PostgreSQL** + **Drizzle ORM** |
| 실시간 | 없음 | **WebSocket** (Socket.IO 또는 Supabase Realtime) |
| 인증 | 없음 | **Supabase Auth** 또는 **Clerk** |
| 결제 | 없음 | **토스페이먼츠** (국내) |
| 이미지 | 없음 | **Cloudflare R2** 또는 **AWS S3** |
| 배포 | 없음 | **Vercel** (Frontend) + **Fly.io** (Backend) |

> **권장:** Supabase 사용 시 Auth + PostgreSQL + Realtime + Storage를 단일 서비스로 처리 가능 → MVP 개발 속도 최대화

### 5-3. 데이터 모델 (핵심)

```sql
stores          -- 매장 (테넌트)
  ├── users     -- 직원 (role: owner/manager/staff)
  ├── tables    -- 테이블
  ├── menus     -- 메뉴 카테고리
  │   └── menu_items  -- 메뉴 아이템
  │       └── option_groups → option_choices
  ├── orders    -- 주문
  │   └── order_items
  └── waitings  -- 대기 고객

subscriptions   -- 구독 플랜 상태
```

### 5-4. URL 구조 변경

| 현재 | SaaS 전환 후 |
|------|------------|
| `/table/:id` | `/m/:storeSlug/:tableId` (고객 메뉴) |
| `/admin` | `/dashboard` (점주 어드민) |
| `/waiting` | `/w/:storeSlug` (대기 키오스크) |
| `/` | `/` (마케팅 랜딩) |
| 없음 | `/auth/login`, `/auth/signup` |
| 없음 | `/onboarding` |
| 없음 | `/settings` |

## 7. 비기능 요구사항 (Non-Functional)

| 항목 | 목표 |
|------|------|
| **성능** | 고객 메뉴 페이지 LCP ≤ 2.5초 (3G 기준) |
| **가용성** | 99.9% uptime (월 43분 이하 다운타임) |
| **실시간 지연** | 주문 → 대시보드 알림 ≤ 1초 |
| **보안** | HTTPS 필수, XSS/CSRF 방어, PCI DSS 준수 (결제) |
| **확장성** | 매장 10,000개 이상 지원 가능한 멀티테넌트 설계 |
| **모바일** | 고객 메뉴: 모바일 우선 (현재 유지) |

## 8. 성공 지표 (KPIs)

### 비즈니스

- **MRR** (월 반복 매출): Phase 1 완료 3개월 내 100만원
- **유료 전환율**: Free → Pro ≥ 15%
- **고객 유지율**: 월 ≥ 85%

### 제품

- **온보딩 완료율**: 가입 후 QR 첫 생성까지 ≥ 70%
- **주문 성공률**: QR 스캔 → 주문 완료 ≥ 90%
- **대시보드 DAU**: 가입 매장의 ≥ 60% 일 1회 이상 접속
